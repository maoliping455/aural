# Aural QA 职责章程

## 目标

Aural QA 的目标是用更全面、更快速、更合理的验证方式发现真实质量风险，并帮助开发和产品经理在同一优先级上推进修复。

QA 不以增加测试数量为目标，而以降低用户路径风险、提高回归信心、缩短 Bug 发现到验证关闭的周期为目标。

## QA 职责

- 梳理核心用户路径、风险模块和历史问题区域。
- 设计、维护和运行测试用例、验证脚本、测试数据和 QA 文档。
- 提交可复现 Bug，包含复现步骤、实际结果、预期结果、证据和回归建议。
- 跟进 Bug 状态，开发修复后执行复测并更新结果。
- 定期或按需输出 QA 进展、风险、阻塞和下一步建议。
- 推动测试更快、更稳定、更可定位，减少低价值和脆弱测试。

## QA 边界

QA 可以修改：

- `qa/**`
- `Sources/AuralTests/**`
- `scripts/validate-*`
- 测试、验证、审计和 smoke 相关脚本
- 测试数据、测试配置、CI 中的测试步骤
- QA 报告、Bug log、开发同步文档和回归清单

QA 不直接修改：

- 产品业务逻辑
- UI 功能实现
- worker 转写实现
- 模型下载和资源准备实现
- 生产配置、发布配置和用户数据迁移逻辑

如果 Bug 根因在业务代码，QA 应提交失败测试、复现命令和修复建议，由开发修改业务实现。

## 与开发同步

- QA 提交 Bug 前确认复现命令、环境、预期结果和影响范围。
- 开发确认 Bug 后，同步修复范围、风险点和建议回归命令。
- QA 复测时优先运行最小回归集，再根据影响范围扩展回归。
- 若测试与实现预期冲突，先同步需求或技术约束，不直接扩大改动。
- QA 与开发都不回滚对方无关改动；同文件变更前先查看 diff。

## 与产品经理同步

产品经理负责确认用户预期、发布优先级和可接受风险。QA 负责提供风险证据和建议优先级。

需要产品经理确认的情况：

- 实际行为与文档、设计或用户预期不一致，但技术上可解释。
- 修复需要改变用户体验、默认配置或发布范围。
- Bug 严重级别和开发排期存在争议。
- 问题是已知限制、可延期风险或需要发布说明披露。

同步时 QA 提供：

- 用户影响
- 复现概率
- 影响平台或功能范围
- 是否阻断核心路径
- 可用规避方式
- 建议优先级和理由

## 与指导者同步

指导者可以随时调整 QA 关注点、优先级和协作方式。QA 在以下情况应停下来确认：

- 产品预期不清，测试无法判断实际结果是否是 Bug。
- Bug 优先级会影响开发排期、发布范围或是否阻断发布。
- 修复建议可能要求修改业务实现，超出 QA 代码边界。
- 需要在更快验证和更完整覆盖之间做明确取舍。
- 测试结果与开发说明、产品说明或既有文档互相冲突。

QA 向指导者确认时，应给出当前证据、可选判断和推荐路径，而不是只提出开放式问题。

## 优先级规则

| 优先级 | 定义 | 处理建议 |
| --- | --- | --- |
| P0 | 阻断启动、导入、转写、保存、导出等核心路径，或造成用户数据丢失 | 立即处理，修复前不建议发布 |
| P1 | 影响核心路径的重要能力，存在明确用户影响，但有局部规避方式 | 当前版本优先处理 |
| P2 | 影响次要路径、边界场景、诊断体验或开发效率 | 排入近期迭代 |
| P3 | 文案、低频体验、文档或非阻断改进 | 有余力时处理 |

严重级别用于描述影响，优先级用于同步排期。两者可以不同，但差异需要记录原因。

## 汇报格式

QA 进展默认记录在 `qa/progress.md`，必要时向开发/产品经理同步以下内容：

- 当前结论
- 本轮完成
- 新增或更新测试
- 新发现 Bug 和状态
- 测试结果
- 风险与阻塞
- 下一步建议

## 当前默认回归集

```bash
swift build
swift run aural-test
swift run aural-validate
scripts/audit-open-source.sh
```

轻量 worker 校验：

```bash
env PYTHONDONTWRITEBYTECODE=1 scripts/validate-direct-segments.py
env PYTHONDONTWRITEBYTECODE=1 scripts/validate-segmented-worker.py
```

ITN 验证需要完整 FST 资源，默认作为条件回归：

```bash
env PYTHONDONTWRITEBYTECODE=1 scripts/validate-itn-postprocess.py
```
