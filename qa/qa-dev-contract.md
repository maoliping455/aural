# Aural QA 与开发规约

本规约用于 QA 和开发日常协作；QA 总体职责、边界和产品经理优先级同步机制见 `qa/qa-charter.md`。

## QA 职责

- 只修改测试、验证脚本、测试数据、QA 文档和测试配置。
- 用可复现命令、失败测试、日志和最小样例证明问题。
- 维护 Bug 状态、验证结果和回归建议。
- 优先覆盖核心路径、历史风险、边界输入、错误处理、资源准备和导出行为。
- 和开发保持同一目标：尽快确认真实风险、降低回归概率、保护用户体验。

## 开发职责

- 分析并修复业务代码根因。
- 不通过删除失败测试或降低断言来关闭问题。
- 修复后说明改动范围、风险和建议回归命令。
- 对争议问题补充产品预期或技术约束。

## Bug 状态

- `New`: QA 已记录，等待开发确认。
- `Confirmed`: 开发确认问题存在。
- `In Progress`: 开发处理中。
- `Fixed`: 开发声明已修复。
- `Verified`: QA 复测通过。
- `Rejected`: 非 Bug，需记录原因。
- `Reopened`: 复测仍失败或修复不完整。

## 同步规则

- QA 提交 Bug 前先确认复现命令、环境和预期，避免把需求不清的问题误判为 Bug。
- QA 不直接改业务代码；如果最小复现指向业务实现，只提交失败测试、日志和修复建议。
- 开发修复前后应同步影响范围，QA 按影响范围选择最小回归集和必要扩展回归。
- 对争议问题先同步产品预期和用户影响，再决定 `Confirmed` 或 `Rejected`。
- 涉及用户体验、发布范围、默认配置或延期风险的问题，需要产品经理确认优先级。
- QA 和开发都不回滚对方无关改动；同文件改动必须先看 diff，再做最小范围变更。

## 提交 Bug 必填字段

- 标题、严重级别、优先级、状态。
- 影响范围和环境信息。
- 复现步骤、实际结果、预期结果。
- 证据路径或命令输出摘要。
- 最小复现用例和修复后回归建议。
